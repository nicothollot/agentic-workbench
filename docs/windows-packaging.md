# Local Run And Windows Packaging

For personal use on the same machine, run the app from the repo instead of creating a new Windows executable:

```powershell
npm run build
npm start
```

`npm run build` compiles the local Electron assets. `npm start` launches those assets through the locally installed Electron runtime.

The project also has packaging commands for producing shareable desktop artifacts from the current source tree:

```bash
npm run package:win
npm run package:mac
```

What the commands do:

- run the normal production compile step
- regenerate the build icon assets from `assets/branding/interface_icon.png`
- stages a minimal packaged-app directory so `electron-builder` does not depend on the full dev workspace layout
- packages an unpacked Windows app folder or macOS disk image with `electron-builder`
- replaces the generated Windows app executable with the unmodified official Electron Windows executable bytes
- copies the final distributable into the detected Downloads folder

Default target selection:

- `npm run build` only compiles the local app assets and does not package a distributable artifact.
- WSL/Linux and native Windows packaging commands build an unpacked Windows app folder by default.
- `npm run package:win` requests the unpacked Windows app folder.
- `npm run package:win:portable` requests the older one-file portable `.exe`.
- `npm run package:mac` requests the macOS `.dmg` and must be run on macOS.

## Output Location

Final distributables are copied to the detected Downloads folder. In WSL, the script first asks Windows for `%USERPROFILE%` and converts that to the mounted WSL path, so this machine resolves to:

```text
/mnt/c/Users/nicot/Downloads
```

Typical final artifacts:

- `Codex Agent Workbench-1.0.0-windows-x64-unpacked/`
- `Codex Agent Workbench-1.0.0-macos-universal.dmg`

Set `AWB_PACKAGE_OUTPUT_DIR=/path/to/output` to override the destination. `AWB_DOWNLOADS_DIR` is also supported for compatibility.

Internal staging output is written under `.electron-builder/` and can be deleted safely after packaging.

## Windows Executable Notes

Smart App Control can block newly built or unfamiliar `.exe` files. It does not infer that a file is for personal use from product names, repository names, or documentation. A code-signing certificate and reputation are the normal way to reduce this class of warning, but local packaging intentionally does not use signing credentials.

For local Windows folder builds, the packaging script now keeps the app resources generated by Electron Builder but restores the executable itself from the official Electron Windows zip. This avoids Electron Builder resource edits to the `.exe` and preserves the upstream Electron executable bytes. It is still not a substitute for publisher signing and Windows reputation, but it is the least invasive local packaging path.

If you do not need a packaged Windows app, use the local workflow:

```bash
npm run build
npm start
```

The default Windows packaging command creates an unpacked app folder instead of a one-file self-extracting executable:

```bash
npm run package:win
```

The portable `.exe` target is still available for compatibility, but it is more likely to trigger Windows reputation checks:

```bash
npm run package:win:portable
```

The packaging script does not request extra files, certificates, signatures, or secrets for Windows builds.

## How To Launch Or Share

Windows:

- For personal use from the repo without packaging, run `npm run build` and then `npm start`.
- Open the generated unpacked folder and run `Codex Agent Workbench.exe`.
- This is a native Windows Electron build. It does not depend on the Linux Electron runtime at launch.

macOS:

- Send or open the generated `.dmg`.
- The current developer build is not notarized, so macOS Gatekeeper may warn.

The app can still use WSL for Git, Codex, and repository operations on Windows, which is the intended architecture.

## What To Compare Against WSL Dev Mode

Compare the packaged Windows app against `npm run dev` in WSL for:

- startup time
- window open/maximize smoothness
- sidebar and large-list scrolling smoothness
- tab or pane switching latency
- resize responsiveness
- general input latency while the UI is busy

Keep the same repository loaded in both cases so the comparison stays meaningful.

## Notes

- Packaging from WSL may need outbound access so `electron-builder` can fetch Windows Electron artifacts the first time.
- The default Windows package uses Electron Builder's `dir` target for an unpacked app folder.
- Windows executable resource editing is disabled for local developer exports so WSL builds do not require Wine, and the generated app executable is replaced with the unmodified official Electron Windows executable.
- The macOS package uses a universal `.dmg` target and must be built on macOS.
- `npm run dist:win` builds the unpacked Windows app folder; `npm run dist:win:portable` remains available for the portable `.exe`.
